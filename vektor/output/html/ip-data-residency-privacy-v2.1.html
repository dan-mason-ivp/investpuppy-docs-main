May 2026 · Version 2.1 · Draft — pending legal counsel review in each jurisdiction

No position in this document should be relied upon as a definitive legal position without confirmation by qualified counsel in the relevant jurisdiction.

Scope: this framework applies to both Vektor : Boutique and Vektor : Institutional. Privacy law obligations exist for both channels. The architecture and the level of obligation differ. This document covers both.

Privacy law applies to BOTH channels. What differs is the compliance architecture and the level of obligation.

Vektor : Boutique — InvestPuppy holds PII. InvestPuppy is a data processor. Standard DPA required with every client.

Vektor : Institutional (hybrid) — InvestPuppy holds only pseudonymised data. PII stays in client's identity vault. DPA obligations materially reduced.

This document covers both compliance postures. Do not treat privacy as an institutional-only concern.

1 · Purpose and Scope

The Vektor platform processes financial data on behalf of wealth management professionals. As the platform distributes across Singapore, Switzerland, the UK, and the Middle East—through both the Vektor : Boutique direct channel and the Vektor : Institutional licensing channel—data residency and privacy requirements are material compliance and commercial considerations for both channels.

This document addresses:

  • What data the platform holds and processes, classified at the attribute level
  • What the applicable privacy and data residency requirements are in each target jurisdiction
  • The compliance posture for Vektor : Boutique and for Vektor : Institutional
  • What the development team must do to implement these frameworks

2.2 Public Reference Data

All tables in the market_data schema contain publicly available information: ticker symbols, ISINs, OHLCV price data, exchange rates, and risk-free rates. None is personal data. Cloud-hosted without qualification.

2.3 Proprietary Business Data

strategy_master and strategy_instruments contain investment methodology configuration—instrument selections, weights, indicator assignments, backtest results. This is the manager's IP. Not personal data. Protection is contractual. Cloud-hosted.

Open question for legal review: can InvestPuppy use aggregated, anonymised strategy data across clients to improve the ML model or publish research? The subscription agreement must explicitly permit or prohibit this.

2.4 Pseudonymised Operational Data

The operational tables contain portfolio holdings, transactions, and orders keyed to UUID identifiers. Under GDPR Article 4(5), pseudonymisation means data can no longer be attributed to a specific data subject without additional information kept separately. In the hybrid model, the UUID-to-identity mapping is held exclusively in the client's on-premises identity vault. InvestPuppy's cloud holds only UUIDs.

The Breyer principle (CJEU C-582/14) applies to GDPR-derived frameworks. Singapore courts are not bound by EU precedent—applicability to PDPA requires independent Singapore counsel confirmation. The principle applies to the hybrid model only. In the standard cloud model (Boutique), InvestPuppy holds both UUIDs and PII—the pseudonymisation argument does not apply.
Aggregate re-identification risk: combinations of operational data may allow inference about the underlying individual (the 'mosaic effect'). Risk is low in a large user base but should be considered below 50 active portfolios.

2.5 Personal Identifying Data (PII Only)

  • customer_name — the name of the client or entity
  • Address fields (if implemented)
  • Date of birth (if stored)
  • National identification number, passport number, or equivalent
  • Contact details: phone number, email address
  • portfolio_name — if it contains client-identifying terms (e.g. 'Chen Family Growth'). Generic names are not PII.

2.6 Boundary Cases Requiring Legal Opinion

⚠ kyc_status: values include 'rejected' which reveals compliance risk characteristics of an individual. Some frameworks may classify this as sensitive data. Treat as PSEUDO pending legal confirmation.
⚠ risk_profile: values are conservative/moderate/aggressive financial profiling data. GDPR Recital 71 identifies profiling as a specific concern. May warrant additional safeguards.
⚠ broker_order_id: IBKR order references may be traceable to the real client account if InvestPuppy is compelled to disclose them. Potential re-identification path. Legal assessment required.

3 · The Identity Vault Architecture (Hybrid Model)

The identity vault is the on-premises component of the Vektor : Institutional hybrid deployment. It holds the UUID-to-PII attribute mapping and is the only system that knows the real identity behind a UUID. The following applies only to the hybrid deployment. In the standard cloud model (Boutique), InvestPuppy holds PII directly.

3.1 Concept

The vault holds one data type: UUID → {customer_name, address, DOB, national_ID, contact_details}. It does not hold portfolio positions, market data, or operational data. The on-premises footprint is minimal—a purpose-specific key-value store.

3.2 Data Flow

  1. Client onboarding: UUID generated. UUID sent to Vektor cloud. UUID-to-PII mapping stored only in on-premises vault.
  2. Platform operation: Vektor cloud processes portfolio data using UUIDs only. InvestPuppy's cloud never requests real client identity.
  3. Human display: Admin portal queries the identity vault to resolve UUIDs to real names. Resolution happens within the client's infrastructure.
  4. Reporting: Client reports assembled by combining pseudonymised data from cloud with identity resolution from vault, within the client's infrastructure.
  5. Termination: Vault entries are the client's responsibility. InvestPuppy's cloud data deleted within 30 days of termination.

3.3 Minimum Vault Security Specification

3.4 The Claim It Enables

"No InvestPuppy system ever processes your clients' names, addresses, or identifying information. Your client identities never leave your infrastructure. What we process in the cloud is pseudonymised portfolio data and public market data. The link between our reference codes and your real clients exists only in a system you control."

This claim is structural, not policy-based. It cannot be violated by a configuration error or security breach on InvestPuppy's side. A breach of InvestPuppy's cloud infrastructure does not result in personal data exposure and therefore may not trigger regulatory breach notification obligations or associated fines.

4.1 Vektor : Boutique — Standard Cloud Compliance Requirements

The following are compliance requirements—not options—before the Vektor : Boutique channel distributes in any jurisdiction.

  • Data Processing Agreement (DPA). Required with every boutique subscriber. InvestPuppy as processor, client as controller. Must cover: processing purposes, data categories, security measures, sub-processors, data subject rights, breach notification, and termination.
  • Sub-processor disclosure. InvestPuppy uses AWS (cloud infrastructure) and AWS SageMaker (ML inference) as sub-processors. The DPA must name these and the client must consent.
  • Regional cloud deployment. Singapore clients: AWS ap-southeast-1. Swiss clients: eu-central-2 (Zurich). UK clients: eu-west-2 (London).
  • Data breach notification. InvestPuppy must notify the supervisory authority and affected clients if its cloud is breached and PII is involved. Timelines: 72 hours (UK GDPR, nDSG); 3 days (PDPA Singapore).
  • Data subject rights. InvestPuppy must assist the data controller in responding to subject access, correction, and deletion requests within prescribed timeframes.
  • Data retention. PII may not be retained beyond the period necessary. Subscription termination triggers a data deletion obligation.
  • Data Protection Officer. GDPR, nDSG, and other frameworks require or recommend a DPO as InvestPuppy grows. Not required at launch but should be in the compliance roadmap.
⚡ Action: The DPA template for Vektor : Boutique is a missing document. It must be produced, reviewed by Singapore and Swiss counsel as first-priority jurisdictions, and signed by every client before they subscribe. This is a pre-distribution requirement.

5.1 Singapore — PDPA 2012 (amended 2021)

The PDPA's broader definition — data identifiable "from that data and other information to which the organisation has or is likely to have access" — may affect the pseudonymisation argument. Singapore counsel confirmation is specifically required against this definition.

2021 amendments: mandatory breach notification (3 calendar days), enhanced penalties (up to SGD 1 million or 10% of annual turnover). Under the pseudonymised model (Institutional), InvestPuppy's MAS TRM vendor classification shifts from 'personal data processor' to 'market data and computation service provider.'

5.2 Switzerland — nDSG (September 2023)

The nDSG explicitly defines pseudonymisation at Article 5(e). Aligns closely with GDPR. The Breyer principle should apply; Swiss counsel confirmation required. Note: nDSG article numbering should be verified against the enacted text—earlier drafts differed.

The nDSG identifiability test has not been extensively litigated in Swiss courts. Its scope relative to GDPR is uncertain. Swiss counsel should specifically address whether the identifiability test produces the same result as GDPR on the pseudonymisation position.

Bankgeheimnis (banking secrecy) is a professional obligation beyond privacy law. The identity vault model satisfies this: client identities remain exclusively in the IAM's control. Bankgeheimnis also applies to Vektor : Boutique — the DPA for Swiss boutique clients must specifically address banking secrecy obligations. The IAM's supervisory organisation (AO) will typically review the DPA as part of ongoing supervision. FINMA's guidance on cloud use requires the supervised entity to retain control over its data — the hybrid model directly satisfies this.

5.3 United Kingdom — UK GDPR and DPA 2018

UK GDPR Art.4(1) and Art.4(5) are identical to EU GDPR. The Breyer principle applies. Singapore's UK adequacy status requires confirmation with UK counsel before transfers of UK-subject personal data to Singapore are relied upon. Where adequacy is absent, IDTAs (International Data Transfer Agreements) are required.

FCA requirements: PS21/3 (operational resilience), SYSC 8 (outsourcing), SYSC 13 (operational risk) all apply to FCA-regulated boutique clients. UK distribution remains blocked pending the UK-01 FCA legal opinion. Note: the UK-01 FCA opinion and the UK DPA template must be coordinated — the DPA for UK boutique clients must address both UK GDPR and FCA SYSC 8/13 requirements.

5.4 UAE — DIFC

DIFC DPL is explicitly modelled on GDPR. Pseudonymisation defined equivalently to GDPR Art.4(5). Breyer principle should apply; DIFC counsel confirmation required. AWS me-central-1 is the UAE-specific region (Dubai). AWS ME-SOUTH-1 is in Bahrain and does not provide UAE data residency.

Cultural context: Client confidentiality in Gulf wealth management exceeds regulatory requirements. A family office principal in Dubai (DIFC-regulated) or Abu Dhabi (ADGM-regulated) expects an unqualified answer to "Can your staff see my client names?" The hybrid model provides that answer.

5.5 Saudi Arabia — PDPL and SAMA

The PDPL imposes strict localisation: personal data of Saudi nationals must be stored in Saudi Arabia. The identity vault model does not resolve this — the PDPL applies to all personal data about Saudi nationals, potentially including pseudonymised data. Saudi Arabia is not in the near-term target market. Becomes viable when the AWS Saudi Arabia region (me-central-1 KSA) is operational.

9 · Action Summary for Development Team

If you are reading this document as a developer, start here before reading the full analysis. Complete all P1 items before any institutional deployment begins.

P1 — Critical: Before Any Institutional Deployment

  1. Audit event_data JSONB. Check every event_data entry type in order_events for PII content. If any contain customer or portfolio names as text, this is a live PII exposure requiring immediate remediation.
  2. Audit error_message field. Check the orders table error_message field for PII content. Remediate if PII found.
  3. Audit all AWS service outputs. Verify that no SageMaker inference call, CloudWatch log, SQS message, SNS payload, or EventBridge event includes customer_id, portfolio_id, or any PII attribute.
  4. Audit Slack notifications. Confirm whether trade signal, order status, or portfolio activity notifications contain customer names or other PII.
  5. Produce DPA template. The Vektor : Boutique standard cloud model cannot distribute without a Data Processing Agreement. Commission Singapore and Swiss counsel to produce and review a standard template.

P2 — Required: Before Production Go-Live

  1. Add classification column to DATA-MODEL-overview. PUB / PROP / PSEUDO / PII / BOUNDARY for every field. Resolve BOUNDARY cases with counsel.
  2. Design the identity vault architecture. Resolve UUID generation, vault technology, and vault API interface. Document minimum security specification compliance.
  3. Design admin portal for hybrid model. Resolve name display architecture for the hybrid deployment.
  4. Define portfolio_name convention. Enforce non-identifying naming or move to vault.
  5. Address broker_order_id classification. Commission legal opinion per jurisdiction.
  6. Define new field governance process. Classification decision workflow before any new field enters production.

P3 — Design Decisions

  1. Resolve strategy data aggregate use. Decision and subscription agreement clause.
  2. Encryption at rest for pseudonymised data. Confirm as best practice regardless of legal requirement.
